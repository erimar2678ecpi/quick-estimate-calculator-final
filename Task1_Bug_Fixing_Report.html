Task 1: Bug Fixing and Debugging Report

Quick Estimate Calculator React Project

Introduction

The bug fixing process is the structured method used to identify, diagnose, correct, test, and document issues found during software testing. A good bug fixing process starts by reproducing the problem, isolating the cause, applying a focused fix, and then retesting the application to confirm that the issue is resolved without creating new problems. This project used that process to improve both functionality and presentation quality.

Bugs and Issues Identified During Testing

  1. Font size was too small. Several labels, helper areas, buttons, and list entries were difficult to read, especially for a final presentation.
  2. Stale estimate data remained visible. After calculating an estimate, changing the service hours did not immediately clear the old total, which could confuse users.
  3. Duplicate saved estimates were possible. Users could press the save button repeatedly and create repeated history entries for the same estimate.
  4. Reset behavior was too aggressive. Resetting the form also cleared all saved history, even when the user only wanted to correct the current estimate.
  5. Input accessibility and validation needed improvement. Labels were not strongly connected to the input fields, and helper/error messaging needed clearer behavior for testing and usability.

Debugging Techniques Used

  • Manual testing: The calculator was tested by entering valid, invalid, empty, and repeated values to observe how the interface responded.
  • State tracing: The behavior of the current estimate, hours input, and estimate history was reviewed to determine when the UI became inconsistent.
  • Logging and inspection: Developer tools and component inspection were used to review layout spacing, typography, and state-driven rendering problems.
  • Unit testing: React Testing Library tests were updated so the project now checks rendering, validation, calculation, stale total clearing, and duplicate-save prevention.
  • Responsive design checks: Browser developer tools were used to verify that the larger text and new layout still worked on smaller screens.

Fixes Applied

  • Increased the global font size and rebuilt the layout for better readability.
  • Added a stronger visual hierarchy with a polished hero section and improved cards.
  • Cleared the total estimate whenever the user changes the service hours input.
  • Prevented duplicate saves by disabling the save button after the estimate is stored.
  • Separated Reset Form from Clear History for better user control.
  • Improved validation with clearer error messages and helper text.
  • Added better accessibility through labeled inputs and error associations.

Results

After applying the fixes, the application became easier to read, more visually appealing, and more reliable during normal use. The new design is better suited for the final week presentation, and the logic fixes reduce confusion when users calculate and save multiple estimates.

Conclusion

This task demonstrated the importance of following a bug fixing process during testing. By reproducing issues, analyzing the cause, using debugging tools, and retesting the final result, the project improved in functionality, usability, accessibility, and appearance. The application is now in a better state for deployment and presentation.
